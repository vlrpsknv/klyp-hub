Killer Insight (из свежего ресерча)

Ни одна micro-drama платформа не имеет self-serve creator publishing console. ReelShort, DramaBox, ShortMax, FlexTV, PineDrama — все работают как гейткиперы: контент заказывается, покупается или производится внутренне. Единственный конкурент — Vigloo Studio (Spoon Labs, март 2026), но у них только analytics dashboard, без creation tools.

Clipt сидит в полностью пустом квадранте: open/self-serve platform + AI creation suite. Это YouTube-before-YouTube-Studio момент для micro-drama. Используй эту фразу на онбординге.

01 — Процесс

Первые 90 дней

Что предлагать как дизайн-процесс. Адаптировано под стартап с 4-месячным MVP, dual-product (Console + App), команду в формировании.

Делать

  • Storyboard-first workflow [сначала раскадровка, потом генерация] — UX Console повторяет 10-шаговый алгоритм создания видео
  • Tokens before components [токены — базовые переменные: цвета, отступы, скругления. Сначала они, потом кнопки и карточки]
  • Ship → Systematize [сначала выпускай, потом систематизируй] — первые 2 недели wireframes [каркасные схемы экранов], не дизайн-система
  • Prototype in Figma, validate in browser — hi-fi [высокая детализация] Figma для alignment [согласования с командой], coded prototype [рабочий прототип в коде] для validation [проверки на реальных пользователях]
  • One source of truth [единый источник правды — одно место, где лежит актуальная версия] — Figma Variables = source of truth, Code Connect для sync [синхронизации дизайна с кодом]

Не делать

  • Начинать с дизайн-системы без понимания product flows [пользовательских сценариев]
  • Делать pixel-perfect [идеально точный дизайн до пикселя] до alignment [согласования] по user flows [путям пользователя по экранам]
  • Проектировать оба продукта параллельно — сначала Console (фокус команды)
  • Копировать UI конкурентов — DramaBox/ReelShort не AI-native [изначально построены не вокруг AI]
  • Тратить неделю на buttons — это решения о tokens [переменных дизайн-системы]
Неделя 1-2
Discovery

Погружение и аудит

  • Stakeholder interviews [интервью с ключевыми людьми] — Ярослав (product vision [видение продукта]), Артём (business model, unit economics [экономика одной единицы — сколько стоит/приносит один эпизод]), devs (tech stack [набор технологий], constraints [ограничения])
  • Competitor UX audit [аудит UX конкурентов — разбор сильных/слабых мест их интерфейсов] — скачать ReelShort, DramaBox, LTX Studio, Krea. Записать screenflows [последовательности экранов], отметить паттерны [повторяющиеся решения]
  • User persona workshops [воркшопы по портретам пользователей — кто наш типичный creator] — Solo AI-artist vs micro-studio vs content agency. Разные workflows [рабочие процессы], разные потребности
  • Алгоритм → User Flow [путь пользователя по экранам] — декомпозировать 10-шаговый алгоритм в UX-screens и user journey map [карту пути пользователя]
  • Information Architecture [информационная архитектура — что где лежит, как организованы экраны и навигация] — структура Console: навигация, иерархия, ключевые экраны
Неделя 3-4
Wireframes

Структура и потоки

  • Lo-fi wireframes [черновые каркасы экранов без визуального дизайна — только структура] ключевых экранов Console: Dashboard [главный экран], Project, Storyboard [раскадровка], Scene Editor, Character Studio, Timeline [временная шкала монтажа], Preview, Publish
  • Critical paths [критические пути — самые важные сценарии, без которых продукт не работает] — 3 key flows: New Series → First Episode → Publish; Character Creation → Reuse [переиспользование]; Scene → Generate → Edit → Approve
  • Design sprint [интенсив по методологии Google Ventures: за 3-5 дней от идеи до тестируемого прототипа] на самый рискованный flow
  • Prototype testing [тестирование прототипа] — внутренняя команда проходит flow, фиксируем friction points [точки трения — где пользователь спотыкается]
  • Token foundation [фундамент токенов] — параллельно закладываем primitives [базовые значения]: цвета, spacing scale [шкала отступов], type scale [шкала размеров шрифтов], radius [скругления углов], motion [анимации]
Месяц 2
Visual Design

Визуальный язык и прототипы

  • Visual explorations [визуальные исследования — 2-3 варианта стиля UI]. Рекомендация: Calm AI [спокойный, ненавязчивый AI-интерфейс] (ближе к Linear), не Overwhelming AI [перегруженный опциями] (HeyGen)
  • Hi-fi screens [экраны с высокой детализацией — выглядят как финальный продукт] основных экранов после approve направления
  • 10 core components [базовых компонентов — переиспользуемых элементов интерфейса] — Button, Input, Select, Checkbox, Card, Modal [всплывающее окно], Badge [значок-метка], Avatar, Toast [всплывающее уведомление], Progress
  • Semantic tokens [смысловые токены — переменные, названные по назначению, а не по значению: «цвет-основного-текста» вместо «серый-900»] — surface-primary, text-primary, border-interactive. Separate collections для Console и App
  • Light + Dark modes [светлая и тёмная тема]
  • Dev handoff [передача дизайна в разработку — процесс, где dev получает всё нужное для сборки] первого flow — тестируем, что dev может собрать screen без дополнительных вопросов
Месяц 3
Iteration

Product-specific паттерны и рафинирование

  • Console-specific компоненты: Storyboard Card [карточка сцены в раскадровке], Generation Panel [панель генерации видео], Asset Browser [каталог ресурсов — картинки, персонажи, клипы], Timeline, Character Card, Cost Preview [превью стоимости перед генерацией], Quality Gate [контрольная точка качества — проверка перед публикацией]
  • Streaming App foundation — начинаем wireframes App (feed [лента контента], player [видеоплеер], paywall [экран оплаты, блокирующий контент], creator profile, coin wallet [кошелёк монет])
  • Design QA [проверка качества дизайна — сравнение собранного в коде с макетами] — первые pages собраны в dev, ревью расхождений
  • Documentation — guidelines [руководства по использованию компонентов] на одной странице, Figma Code Connect
  • Internal user testing [внутреннее тестирование на команде] — команда проходит полный flow creation → publish
Месяц 4
MVP

Полировка и запуск

  • Bug fixes и pixel polish [финальная доводка до идеала]
  • Onboarding flow [сценарий первого знакомства пользователя с продуктом] — TTFV < 5 минут [Time to First Value — время до первого «ага, это работает!»]. Creator заходит → видит template → генерирует первый clip за 3 минуты
  • Empty states [пустые состояния — что видит пользователь, когда данных ещё нет] и error handling [обработка ошибок — что показываем, когда что-то пошло не так]
  • Responsive QA [проверка на разных размерах экрана]
  • Launch checklist [чек-лист перед запуском] — Detach test [тест: сторонний дизайнер использует систему, не сломав её], A11y audit [аудит доступности для людей с ограничениями], performance check [проверка скорости загрузки]

Ключевой принцип

Ship first, systematize second. [Сначала выпусти, потом систематизируй.] Для 4-месячного MVP дизайн-система — это не первый deliverable [результат работы, который ты сдаёшь]. Первый deliverable — рабочий user flow [путь пользователя по экранам], который можно тестировать. Система кристаллизуется из решений, принятых для product, а не наоборот.

01b — Методология

Что я предлагаю как процесс

Конкретные фреймворки и подходы, которые я принесу в команду. Каждый адаптирован под контекст: стартап, 4 месяца до MVP, два продукта, команда формируется.

01 Compressed Double Diamond [сжатый двойной ромб]

Что это: Классический дизайн-процесс из двух ромбов. Первый ромб — понять проблему. Второй — найти решение. В каждом ромбе две фазы: расширение [собираем варианты, исследуем, генерим идеи] и сужение [фильтруем, приоритизируем, выбираем лучшее].

Почему именно это: Стандарт индустрии (British Design Council), но обычно занимает 3-6 месяцев. Для Clipt я сжимаю до 4 недель на первый diamond [ромб] и 8 недель на второй, не теряя качества решений.

Как это выглядит на практике для Clipt:

  • Ромб 1 — Discover + Define [исследуй + определи] (неделя 1-2): Собираю всю информацию — интервью с командой, аудит конкурентов, разбор алгоритма, портреты пользователей. Потом сужаю: формулирую ключевые проблемы, рисую user flows [пути пользователя], фиксирую scope [объём работы]
  • Ромб 2 — Develop + Deliver [разработай + выпусти] (неделя 3 → месяц 4): Генерирую варианты решений (wireframes, visual explorations), тестирую, сужаю до финального дизайна, передаю в разработку, полирую

Что команда от этого получает: Структурированный процесс вместо хаоса. Каждую неделю понятно, что делаем и почему. Минимум переделок, потому что решения проверены до того, как попали в код.

02 Design Sprint [дизайн-спринт — интенсив от идеи до тестируемого прототипа за 3-5 дней]

Что это: Методология Google Ventures. За 3-5 дней команда проходит полный цикл: понять проблему → набросать решения → выбрать лучшее → сделать прототип → протестировать на реальных людях. Без кода, без месяцев разработки.

Когда использовать в Clipt: Для самых рискованных и непонятных задач. Примеры:

  • "Как должен выглядеть flow генерации видео?" — это самый рискованный сценарий Console, от него зависит всё
  • "Как creator сохраняет и переиспользует персонажа?" — character consistency — это главная техническая проблема, и UX должен её решать
  • "Как выглядит paywall [экран оплаты] в Streaming App?" — от этого зависит монетизация

Формат для стартапа: Сжатый до 3 дней (а не 5). День 1 — карта проблемы + наброски решений. День 2 — прототип в Figma. День 3 — тестируем внутри команды (или на 3-5 внешних людях). По результатам — решение принято, можно строить.

Почему это ценно: Вместо 3 недель споров «как лучше» — 3 дня и ответ, подкреплённый тестом. Экономит время и убивает неопределённость.

03 Jobs-to-be-Done [JTBD — фреймворк «для какой работы пользователь нанимает наш продукт»]

Что это: Вместо того, чтобы думать «какие фичи добавить», думаем «какую задачу пользователь пытается решить». Пользователь не покупает дрель — он покупает дырку в стене.

JTBD для Clipt Publishing Console:

  • Основной Job: «Превратить мою идею в опубликованный эпизод AI-сериала с минимальной болью и предсказуемым качеством»
  • Supporting Jobs [вспомогательные задачи]:
    • «Сохранить внешность персонажа одинаковой через все сцены и эпизоды»
    • «Понять, сколько будет стоить генерация, до того как я нажму кнопку»
    • «Быстро перегенерировать одну плохую сцену, не трогая остальные»
    • «Понять, какой мой эпизод смотрят до конца, а какой бросают»

JTBD для Streaming App:

  • Основной Job: «Найти сериал, который затянет, и досмотреть его до конца»
  • Supporting Jobs: «Вернуться к тому, на чём остановился», «Найти что-то похожее на то, что понравилось», «Не переплачивать за то, что не досмотрю»

Как использовать: Каждая фича и каждый экран оцениваются через JTBD. «Эта фича помогает пользователю выполнить его job? Да → делаем. Нет → в бэклог [очередь задач на потом] или в мусор.»

04 Continuous Discovery [непрерывное исследование — каждую неделю узнаём что-то новое о пользователях]

Что это: Методология Терезы Торрес. Вместо «сделали research раз и забыли» — каждую неделю разговариваем хотя бы с одним пользователем или анализируем данные. Это привычка, не проект.

Когда включать: После запуска MVP. До запуска — работаем с гипотезами и внутренним тестированием.

Как это выглядит:

  • Weekly user touchpoint [еженедельная точка контакта с пользователем] — 15-30 минут, один creator. Не формальное интервью, а наблюдение: «покажи, как ты создаёшь эпизод»
  • Opportunity Solution Tree [дерево возможностей — визуальная карта: цель → возможности → решения → эксперименты] — обновляется каждую неделю
  • Assumption mapping [карта допущений — что мы предполагаем, но не проверили] — какие наши гипотезы самые рискованные? Проверяем их первыми

Почему это важно для Clipt: AI-видео — это новый рынок. Никто не знает, как creator хочет работать с AI-генерацией. Единственный способ узнать — спрашивать и наблюдать систематически, а не гадать.

05 Atomic Design [атомарный дизайн — сборка интерфейса от мельчайших элементов к страницам]

Что это: Методология Брэда Фроста. Интерфейс строится слоями, от простого к сложному:

  • Атомы — мельчайшие элементы: кнопка, иконка, текстовое поле, label [подпись]
  • Молекулы — группа атомов: поле ввода + label + иконка ошибки = search bar [строка поиска]
  • Организмы — группа молекул: header [шапка сайта] = логотип + навигация + кнопка профиля
  • Шаблоны — скелет страницы без реального контента
  • Страницы — финальный экран с реальными данными

Зачем для Clipt: Два продукта (Console + App) должны использовать одни атомы, но собирать из них разные организмы. Кнопка — одна. Но карточка сериала в App и карточка проекта в Console — разные организмы из одних атомов. Это экономит время дизайна и разработки.

На практике: Я строю дизайн-систему именно так — начинаю с tokens [переменных] (это even ниже атомов), потом 10 core components [базовых компонентов] (атомы + молекулы), потом product-specific паттерны [специфичные для продукта организмы].

06 Impact / Effort + Risk Matrix [матрица приоритизации — что делаем первым, а что потом]

Что это: Каждая задача оценивается по трём осям:

  • Impact [влияние] — насколько это важно для пользователя и бизнеса. High = без этого продукт не работает
  • Effort [трудозатраты] — сколько времени займёт дизайн + разработка. Low = день, High = месяц
  • Risk [риск] — насколько велика вероятность, что мы ошибёмся. High = мы не знаем, как это должно работать

Правило приоритизации:

  • High Impact + High Risk → делаем первыми, через design sprint [интенсив]. Пример: flow генерации видео
  • High Impact + Low Risk → делаем быстро, это «low-hanging fruit» [лёгкая победа — результат при минимальных усилиях]. Пример: dashboard [главный экран] со списком проектов
  • Low Impact → после MVP. Пример: кастомные анимации, тёмная тема для Console

Для Clipt прямо сейчас (первые приоритеты):

  • Character consistency UX — high impact, high risk, medium effort. Первый приоритет
  • Generation flow (storyboard → generate → review) — high impact, high risk, high effort. Второй приоритет, через sprint
  • Dashboard + project management — high impact, low risk, low effort. Быстро закрываем
  • Publish flow — high impact, low risk, medium effort. Третий приоритет
07 Progressive Fidelity [постепенное повышение детализации — от набросков к финалу]

Что это: Не рисуем сразу красиво. Повышаем детализацию постепенно, на каждом шаге получая фидбэк.

  • Уровень 1 — Sketch / Paper [набросок на бумаге]: 5 минут. Для себя и для быстрого обсуждения идеи с product owner [владельцем продукта]. Не показываем стейкхолдерам [заинтересованным лицам]
  • Уровень 2 — Lo-fi wireframe [черновой каркас]: 1-2 часа. Серые блоки, текст, стрелки. Для alignment [согласования] структуры и логики с командой. «Правильные ли блоки на экране? Правильный ли порядок?»
  • Уровень 3 — Mid-fi prototype [прототип средней детализации]: 1-2 дня. Кликабельный, с реальной навигацией, но без финальных цветов и стиля. Для тестирования user flows [путей пользователя]
  • Уровень 4 — Hi-fi mockup [детальный макет]: 2-3 дня. Финальный визуал, анимации, states [состояния: hover, active, disabled, error, loading]. Для утверждения и передачи в разработку
  • Уровень 5 — Coded prototype [прототип в коде]: 1-3 дня. Рабочий HTML/React. Для validation [проверки] сложных взаимодействий, которые невозможно протестировать в Figma (video player, drag-and-drop, timeline)

Правило: Никогда не прыгаем с уровня 1 на уровень 4. Каждый уровень — это checkpoint [контрольная точка]: «Мы идём правильно?» Дешевле исправить набросок, чем переделывать финальный макет.

Для видео-платформы: Уровень 5 (coded prototype) особенно важен, потому что ключевые взаимодействия — video playback [воспроизведение видео], timeline scrubbing [перемотка по временной шкале], drag-and-drop [перетаскивание] сцен в storyboard — невозможно проверить в статичном Figma.

08 Component-Driven Development [разработка от компонентов — дизайн и код строятся из одних блоков]

Что это: Дизайнер и разработчик работают с одним и тем же набором компонентов. Кнопка в Figma = кнопка в коде. Карточка в Figma = карточка в коде. Один source of truth [источник правды].

Инструменты:

  • Figma — здесь живут компоненты со всеми variants [вариантами: primary/secondary, small/medium/large, disabled/active] и states [состояниями]
  • Storybook [каталог компонентов в коде — dev видит каждый компонент изолированно с документацией] — зеркало Figma в коде. Каждый компонент живёт отдельно, с примерами и документацией
  • Figma Code Connect [плагин, связывающий компоненты Figma с кодом] — dev кликает на компонент в Figma и видит, какой React-компонент ему соответствует
  • Design Tokens [переменные дизайна] → Style Dictionary [инструмент конвертации] → CSS / Swift / Kotlin — цвета, шрифты, отступы автоматически попадают из Figma в код

Зачем: Разработчик не «на глаз» пытается повторить макет, а использует готовый компонент. Это быстрее, точнее, и поддерживаемо [можно менять в одном месте — обновится везде].

09 Dual-Product Strategy [стратегия для двух продуктов одновременно]

Проблема: Clipt строит два продукта: Console (web, для creators) и Streaming App (mobile, для зрителей). Это два разных мира с разными UX-паттернами [устоявшимися решениями].

Что я предлагаю:

  • Последовательно, не параллельно. Первые 2 месяца — Console (это фокус разработки). App — начинаем wireframes [каркасы] на месяце 3, когда Console уже в коде
  • Shared foundation, different expression [общий фундамент, разное выражение]:
    • Одни tokens [переменные] для базовых значений (цвета, шрифты, размеры)
    • Разные semantic tokens [смысловые переменные] для каждого продукта (Console: светлая, dense [плотная], рабочая. App: тёмная, cinematic [кинематографичная], развлекательная)
    • Разные паттерны взаимодействия (Console: keyboard-first [управление с клавиатуры], table views [табличные представления]. App: gesture-first [управление жестами], swipe [свайп], full-screen video)
  • Одна Figma library [библиотека], два product files. Общие компоненты (кнопки, inputs, модалки) лежат в library. Product-specific [специфичные для продукта] компоненты (Storyboard Card, Video Player Card) лежат в файле продукта
10 Метрики дизайна [как измерить, что дизайн работает]

Что предлагаю отслеживать:

Console (для creators):

  • TTFV [Time to First Value — время до первого «ага!»] — сколько минут от регистрации до первого сгенерированного клипа. Цель: < 5 минут
  • Task completion rate [процент завершения задачи] — какой процент начатых эпизодов доходит до публикации. Если <50% — UX broken [интерфейс мешает]
  • Time per episode [время на эпизод] — сколько часов от идеи до публикации. Цель: 2-4 часа для первого, <1 часа для повторных
  • Generation success rate [процент удачных генераций] — сколько генераций creator оставляет vs перегенерирует. Low = промпт-инженеринг слишком сложный или defaults [настройки по умолчанию] плохие
  • SUS score [System Usability Scale — стандартный опросник юзабилити, 10 вопросов, оценка 0-100] — после первого месяца

Streaming App (для зрителей):

  • Time to first play [время до первого просмотра] — сколько секунд от открытия до начала просмотра. Цель: < 10 секунд
  • Episode completion rate [процент досмотра эпизода до конца] — главный KPI [Key Performance Indicator — ключевой показатель эффективности]. <50% — плохо
  • Episode-to-episode drop-off [отток между эпизодами — какой процент не переходит к следующему] — <15% у хороших серий
  • Paywall conversion [конверсия в оплату на экране paywall] — какой процент платит на каждом paywall point
  • D1 / D7 / D30 retention [процент пользователей, вернувшихся через 1 / 7 / 30 дней]

Как это связано с дизайном: Каждая из этих метрик напрямую зависит от UX-решений. TTFV — от onboarding [сценария первого входа]. Completion rate — от player UI [интерфейса плеера]. Paywall conversion — от того, как выглядит и когда появляется карточка оплаты. Дизайн — это не «красиво», это бизнес-результат.

Constraints — лучший друг дизайнера. 4 месяца до MVP — это не проблема, это фильтр, который убивает всё лишнее и заставляет делать только то, что реально важно.

Как это озвучить на встрече

«Я предлагаю работать в сжатом Double Diamond: первые 2 недели — погружение и структура, потом wireframes и design sprint на самый рискованный flow, потом визуал и компоненты. Параллельно закладываю token-фундамент для дизайн-системы. Дизайн-система кристаллизуется из product decisions, а не наоборот. К концу месяца 2 у нас есть visual direction, 10 core components и первый flow в коде. К MVP — рабочий продукт, который можно тестировать.»

5 критических UX-вызовов Console

01 Character Consistency — первый гражданин, не фича

Проблема: без reference images [эталонных изображений персонажа] / LoRA [метод дообучения AI-модели на конкретного персонажа — «запоминание» внешности] начинается character drift [дрейф персонажа — AI постепенно меняет внешность от сцены к сцене] через 5 сцен. Персонаж меняет лицо, одежду, пропорции.

Решение: Character Studio — dedicated [выделенный, отдельный] экран. Creator создаёт персонажа один раз: 5+ baseline images [базовых изображений] (front, profile, 3/4, emotions), текстовое описание, voice profile [голосовой профиль]. Эти references автоматически подставляются во все сцены.

Референс: Runway Gen-4 References, Kling Element Library, LTX Elements workflow.

UX-детали:

  • Character card visible в storyboard sidebar постоянно
  • Consistency check перед generation — preview overlay персонажа на сцену
  • Version history для character evolution (сезон 1 vs сезон 2)
02 Credit Paranoia — страх тратить кредиты

Проблема: один 8-секундный клип может стоить 15-35% месячных бесплатных кредитов. Пользователь боится экспериментировать.

Решение:

  • Cost preview [предпросмотр стоимости] перед каждой генерацией — "This will use ~5 credits ($0.50)"
  • Draft mode [черновой режим] — быстрая генерация в низком качестве (480p, fewer steps). -60-70% стоимости
  • Batch 4 варианта [пакетная генерация — 4 варианта за один раз] по цене 2.5 — как Midjourney. Выбрать лучший
  • Undo/retry [отмена/повтор] — если result плохой, вернуть 50% credits
03 Storyboard-first vs Prompt-first

Default [по умолчанию]: Storyboard-first [сначала раскадровка]. Пользователь видит 12-20 карточек сцен, каждая с описанием action/emotion/camera. Генерирует по одной сцене, retake per-shot [перегенерация одной конкретной сцены, а не всего видео].

Toggle [переключатель] для pro: Prompt-first mode [режим «сначала текстовый запрос»] — описал весь эпизод текстом, AI разбил на сцены автоматически.

Почему storyboard по умолчанию: LTX Studio установил стандарт. Narrative control [контроль над повествованием]. Retake без перегенерации всего. Cognitive load [когнитивная нагрузка — сколько мозгу нужно обрабатывать] ниже.

04 Calm AI — Console ближе к Linear, не к HeyGen

Проблема конкурентов: Krea — "dense wall of options" [стена настроек — всё на одном экране, не знаешь куда смотреть]. Runway 4.5 — "sidebar packed" [боковая панель забита кнопками]. Overload [перегрузка] на первом экране.

Принцип: Director / Curator [режиссёр / куратор — ты задаёшь направление, AI исполняет], а не кнопочник. UX поддерживает direction, не micromanagement [не ручное управление каждой мелочью].

Конкретно:

  • Smart defaults [умные настройки по умолчанию — всё уже выставлено правильно для типичного случая] + progressive disclosure [постепенное раскрытие — сложные настройки прячутся, пока пользователь сам не захочет их открыть]. Aspect ratio [соотношение сторон], duration, resolution — всё preset [предустановлено] для vertical micro-drama. Менять только если нужно
  • Prompt field first [поле ввода на первом плане — открыл приложение и сразу можешь писать запрос]. Как Perplexity — сразу в поле ввода, без настройки
  • Templates > blank canvas [готовые шаблоны лучше пустого холста]. Зашёл → увидел template (Romance, CEO Revenge, Alpha Werewolf) → собрал пилотный эпизод за 10 минут
  • Guide when stuck, disappear when not [подсказывай, когда застрял — исчезай, когда всё понятно]. Contextual hints [контекстные подсказки] только при пустом state [состоянии экрана] или ошибке
05 Multi-model switching в одном UI

Тренд 2026: пользователь не хочет 10 подписок. Krea, LTX Studio, Freepik Spaces — все конвергируют на multi-model access.

Для Clipt Console: generation panel позволяет выбрать model (Kling / Runway / Veo / Sora) для каждой сцены. Разные модели лучше для разного:

  • Kling 2.6 — длинные клипы (до 2-3 мин), best lip-sync, Element Library
  • Runway Gen-4 — Motion Brush, camera controls, references
  • Veo 3.1 — best realism/physics, native 4K
  • Sora 2 — best narrative intelligence, Cameo

UX: model selector в Generation Panel. Default рекомендация на основе типа сцены (dialogue → Kling, action → Veo, emotional close-up → Runway).

Лучшие AI-продукты не кричат «I'm AI!». Хорошие AI guide, great AI disappear. Kate Syuma, AI UX Analyst
03 — Задать

Умные вопросы для онбординга

Не интервью-вопросы, а вопросы первого рабочего дня. Показывают product thinking, не любопытство.

A Product & Vision
Кто primary user Console — соло AI-artist, micro-студия, или контент-агентство?
Определяет complexity UX. Соло = простой wizard [пошаговый помощник]. Студия = multiplayer [совместная работа нескольких человек], roles [роли: автор, редактор, admin], permissions [права доступа]. Агентство = API-first [приоритет на программный интерфейс для автоматизации], batch [пакетная обработка — много задач разом].
Console — creation tool или management tool? Или оба?
Creation = генерация внутри Console. Management = загрузка готового видео. MVP scope [объём первой версии продукта] вопрос. Скорее всего гибрид, но приоритет важен.
Monetization model для creators? Revenue share, subscription, или pay-per-generation?
Влияет на UX: earnings dashboard, payout page, analytics depth.
Есть ли pitch deck / strategy doc, который можно изучить?
Что говорили инвесторам — это ground truth для product vision.
Какие AI-видео тулы команда уже протестировала? Есть выбранный стек или тестируем с нуля?
Определяет, какие API подключать и какие UX-паттерны generation поддерживать.
B Команда & Процесс
Кто в команде сейчас? Кто выходит параллельно?
Конфигурация day-one. Кто второй дизайнер на Streaming App. Когда планируется расширение дизайн-команды.
Какой tech stack фронтенда? React/Next.js? Mobile — native или cross-platform?
Влияет на design handoff, component library strategy, prototyping approach.
Как организован product process? Sprints? Kanban? Какие тулы (Linear, Notion, Jira)?
Нужно встроиться в existing workflow, а не навязывать свой.
Design review — кто approve? Ярослав? Артём? Или автономия?
Определяет скорость итераций и уровень детализации в презентации решений.
C Brand & Identity
Clipt — финальное название или рабочее? Есть ли уже нейминг, логотип, первые концепты?
Если с нуля — это отдельный workstream. Если есть — нужно получить assets.
Какой тональный регистр бренда? Entertainment (тёплый, яркий) или Tech (минимальный, нейтральный)?
Console — скорее productivity/tech (Linear-like). App — скорее entertainment (яркий, immersive). Два разных визуальных языка.
D Формальные (контракт/оформление)
Критерии прохождения испытательного (3 мес). Конкретные deliverables к 6 июля?
Фиксируем письменно. Метрики, milestones. Это must.
Задачи с «мягкими дедлайнами» к 26-27 апреля — какие именно?
Нужно понять scope до отпуска. Успеть продумать, чтобы не горело.
Доступы: Figma, Notion/Linear, Google Workspace, репозитории — что нужно получить сегодня?
Не уходить с онбординга без доступов. Иначе первый рабочий день = ноль продуктивности.
04 — Ответить

Что спросят — и как отвечать

Типичные вопросы на онбординге Design Lead. Ответы адаптированы под контекст Clipt.

01 «Как ты видишь свою роль здесь?»

Плохо: "Буду рисовать экраны и делать дизайн-систему"

Хорошо: "Моя задача — перевести product vision [видение продукта] в интерфейсы, которые работают. Первые 2 недели — погружение: понять пользователей, алгоритм, технические ограничения. Потом — wireframes ключевых flows Console. К концу месяца — visual direction [визуальное направление — как продукт будет выглядеть] и первые компоненты. Параллельно выстраиваю процесс, чтобы дизайн не был bottleneck [узким горлышком, тормозящим команду] для разработки."

02 «Какой дизайн-процесс ты используешь?»

Ответ: "Зависит от стадии. Сейчас, pre-MVP — Double Diamond [«двойной ромб» — дизайн-процесс из 4 фаз: расширение → сужение → расширение → сужение] adapted для speed:

  • Discover [исследуй] (неделя 1-2): stakeholder interviews, competitor audit [разбор конкурентов], user persona [портрет пользователя]. Не рисую, пока не пойму.
  • Define [определи] (неделя 2-3): IA [информационная архитектура], user flows, lo-fi wireframes [черновые каркасы]. Alignment [согласование] с product и dev.
  • Develop [разработай] (месяц 2): visual explorations, hi-fi [детальные макеты], components. Design sprint на riskiest flow [самый рискованный сценарий].
  • Deliver [выпусти] (месяц 3-4): handoff [передача в разработку], QA, polish [полировка], iteration [итерация — улучшение по результатам] по feedback."

"После MVP — continuous discovery [непрерывное исследование — каждую неделю узнаём что-то новое о пользователях]: weekly user insights → design → ship → measure."

03 «Как приоритизируешь задачи?»

Ответ: "Impact / Effort matrix [матрица «влияние / трудозатраты» — оцениваем каждую задачу по двум осям], но с одним нюансом — для MVP я добавляю ось Risk [риска]. Что может провалить запуск? Character consistency UX — high risk, high impact, medium effort. Это первый приоритет. Красивые анимации — low risk, medium impact, high effort. Это после MVP."

"Для ежедневной приоритизации — Jobs-to-be-Done [JTBD — фреймворк: какую «работу» пользователь «нанимает» наш продукт выполнить]. Console: «превратить мою идею в опубликованный эпизод с минимальной болью». Каждая фича оценивается по тому, насколько она приближает к этому."

04 «С чего начнёшь первую неделю?»

Ответ:

  • День 1-2: Собрать все inputs — pitch deck, алгоритм, tech constraints, существующие наработки. Настроить Figma workspace.
  • День 3: Декомпозировать 10-шаговый алгоритм в user journey map. Каждый шаг = экран.
  • День 4-5: Lo-fi wireframes первого flow — "New Series → First Episode → Publish". Показать команде для раннего feedback.

"К концу первой недели у вас будет IA Console и первый prototype для обсуждения."

05 «Нужна ли нам дизайн-система?»

Ответ: "Да, но не сейчас как отдельный проект. Дизайн-система кристаллизуется из product decisions. Сначала я приму решения о цветах, типографике, spacing через реальные экраны. Потом вычленю паттерны и систематизирую."

"Конкретно: месяц 1 — tokens (primitive + semantic). Месяц 2 — 10 core components. Месяц 3 — product-specific паттерны. К MVP у нас будет рабочая система, не абстрактная библиотека."

06 «Что ты знаешь про AI-video generation?»

Ответ: "Работал с генерацией на Flashyr (face consistency через Leonardo.ai, Seed Dream). Знаком с ландшафтом: Runway Gen-4 — лучший motion control, Kling 2.6 — longest clips + Element Library для consistency, Sora 2 — лучшая narrative intelligence, Veo 3.1 — best physics. Для micro-drama production character consistency — главный challenge, и сейчас это решается на 80% через references / LoRA / Element Libraries."

"Прочитал ваш алгоритм. Core rules (8 сек / 12-20 сцен / close-ups) — это выверенный базис. У меня есть вопросы по character consistency across seasons, reference library и quality gates — хочу понять, как вы планируете решать."

07 «Как работаешь с разработчиками?»

Ответ: "Три принципа:

  • Early involvement. [Ранее вовлечение] Dev видит wireframes, а не готовый дизайн. Фидбэк на уровне 'это сложно реализовать' — до того, как я потрачу неделю на pixel-perfect.
  • Shared language. [Общий язык] Я пишу code (React, Next.js через Claude Code). Понимаю constraints [ограничения]: rendering strategies [стратегии рендеринга — как страница формируется для пользователя], performance, a11y [accessibility — доступность]. Общаемся на одном языке.
  • Design tokens = contract. [Токены = контракт между дизайном и кодом] Figma Variables → Style Dictionary → CSS/iOS/Android. Один source of truth, не PDF спека."
08 «Как относишься к сжатым срокам?»

Ответ: "4 месяца до MVP — это не проблема, это constraint [ограничение]. Constraints — лучший друг дизайнера. Они убивают scope creep [расползание объёма — когда задача незаметно разрастается] и заставляют принимать решения."

"Конкретно: я не буду рисовать 50 экранов. Я найду 5 critical paths [критических путей], отполирую их, остальное — wireframe [каркас] + component library [библиотека компонентов]. MVP — это не 'красиво', это 'работает и не отталкивает'."

07 — Дизайн-система

Token Architecture [архитектура дизайн-токенов — переменных, которые хранят все визуальные решения]

Три яруса. Standard в 2026 — не опция, а ожидание.

Primitives

Raw values [сырые значения]

color-blue-500, spacing-4, radius-sm. Никогда не применяются напрямую в компонентах. Это как палитра красок — сами по себе не используются, но из них собираются осмысленные значения.

Semantic

Purpose-driven [по назначению]

text-primary [цвет основного текста], surface-elevated [фон приподнятого элемента], border-interactive [обводка интерактивного элемента]. Ссылаются на primitives. Это то, что использует дизайнер.

Component

v2 (not MVP)

button-primary-bg [фон основной кнопки]. Опциональный для v1. Добавлять когда система вырастет до 30+ компонентов.

Dual-product token strategy

Shared Primitives
Один collection — базовые цвета, spacing scale, radius scale, type scale. Одинаковые для Console и App.
Console Semantic
Productivity tool [рабочий инструмент]: light-first [светлая тема основная], density-dense [плотная компоновка — много информации на экране], neutral palette + accent [нейтральные цвета + один яркий акцент]. Ближе к Linear/Notion.
App Semantic
Entertainment [развлечение]: dark-first [тёмная тема основная], cinematic [кинематографичный], bold palette [яркие насыщенные цвета]. Ближе к Netflix/DramaBox.
Modes [режимы]
Light / Dark / High contrast [повышенный контраст для доступности] (минимум). Brands / breakpoints mode [режим адаптации под разные бренды / размеры экрана] (nice-to-have [было бы хорошо, но не обязательно]).

08 — Тех. контекст

Technical Awareness

Lead Product Designer не пишет код (ну, почти). Но понимание стека — это язык общения с devs и основа реалистичных решений.

A UI Architecture
Atomic Design
Компоненты от атомов (icon, label) к молекулам (input + label) к организмам (form) к шаблонам к страницам. Помогает structure design system.
FSD
Feature-Sliced Design. Фронтенд-архитектура по бизнес-сущностям. Код организован по фичам, не по типам файлов.
A11y
Accessibility. Семантический HTML (main, nav, section vs div), ARIA-атрибуты (aria-labelledby, aria-live). Console — web-app, A11y baseline нужен.
B Performance Metrics (Core Web Vitals)
LCP
Largest Contentful Paint. Время до появления главного элемента. Для Console — dashboard load. Для App — first video frame. Target: <2.5s.
INP
Interaction to Next Paint. Отклик UI на клик/ввод. Критично для Console — generation panel должен реагировать instant. Target: <200ms.
CLS
Cumulative Layout Shift. Стабильность верстки. Нет сдвигов при загрузке. Для Streaming App — video player не должен прыгать. Target: <0.1.
FCP / TTFB
First Contentful Paint / Time to First Byte. Скорость первой отрисовки и ответа сервера.

Зачем знать: эти метрики влияют на SEO (Streaming App marketing pages), конверсию и retention. Дизайн-решения напрямую влияют — размер изображений, lazy loading, skeleton screens.

C Frameworks & Rendering
React + Next.js
Вероятный стек Console (web). Next.js даёт SSR/SSG из коробки. React ecosystem для компонентов.
TypeScript
Статическая типизация. Меньше ошибок. Предсказуемость изменений.
SSR vs CSR vs SSG
SSR — сервер рендерит HTML (для SEO-страниц: marketing, show pages). CSR — клиент рендерит (для Console dashboard). SSG — статика (для блога, документации). Для Console — SSR для landing, CSR для app.
React Native / Flutter
Cross-platform для Streaming App. React Native — если web team сильная. Flutter — если нужна uniform UI. Уточнить на онбординге.
D State & Data
State Managers
Zustand (modern, simple), Redux Toolkit (classic), MobX. Хранят состояние UI: что в проекте, какие settings, что загружено.
TanStack Query
Серверные данные: кэширование, запросы. Для Console — списки проектов, assets, analytics.
API / REST / GraphQL
Контракт frontend ↔ backend. Console будет делать много запросов к AI APIs — important for loading states UX.
E Quality & Monitoring
Sentry
Error tracking. Автоматический отлов ошибок в production. Дизайнеру важно: error states в UI должны быть спроектированы.
Lighthouse
Аудит performance, a11y, SEO. Запускать на каждый major release. Baseline score — зафиксировать в первую неделю.
WebP / AVIF
Оптимизированные форматы изображений. Вместо PNG/JPEG. Для Console — thumbnails, previews. CDN для доставки.
09 — Vocabulary

Словарь micro-drama

Термины, которые нужно знать и использовать в разговорах. Показывают, что ты в теме.

Vertical drama
Сериал в вертикальном формате 9:16, эпизоды 60-180 секунд. Не нарезанный long-form, а отдельный формат.
Micro-drama
= Vertical drama. Термин из Китая (duanju, 短剧). $11B market 2025.
Paywall point
Эпизод, на котором включается оплата. Обычно на клиффхэнгере. Standard: ep 4, 8, 13.
Coin-based unlock
Доминирующая модель монетизации. Пользователь покупает coins ($5-50), тратит на unlock (10-60 coins/ep ≈ $0.10-0.60).
ARPU / RPD
Average Revenue Per User / Revenue Per Download. ReelShort RPD ~$2.59, DramaBox ~$1.60.
Completion rate
Процент досмотра эпизода. Главный KPI. <50% — переписывают последние 15 сек.
Episode drop-off
Процент ухода между эпизодами. <15% ep1→ep2 у успешных серий.
Cliffhanger
Обрыв на пике эмоций. Ключевой retention hook. Taxonomy: identity reveal, betrayal, impossible choice, secret pregnancy.
Character consistency
Сохранение внешности персонажа через scenes/episodes. #1 technical challenge в AI video. Решается через References, LoRA, Element Library.
LoRA
Low-Rank Adaptation. Метод fine-tuning моделей для запоминания конкретного персонажа.
Reference images
Базовые изображения персонажа (front, profile, 3/4), которые подаются модели для consistency.
Element Library
Kling O1 feature. Сохранённые characters, objects, locations для reuse через scenes.
Motion Brush
Runway feature. Рисовать направление движения на конкретных частях frame [кадра]. Уникальный competitive moat [конкурентный ров — преимущество, которое сложно повторить].
Retake per-shot
Перегенерация одной конкретной сцены, не всего видео целиком. LTX Studio pattern [паттерн — устоявшееся решение]. Must-have [обязательная функция] для Console.
TTFV
Time to First Value. <5 минут для AI tool. <10 секунд для Streaming App (first play).
Storyboard-first
Workflow: сначала разбить на сцены, потом генерировать. Vs prompt-first (описать целиком). Default для Console.
Creator profile
Страница автора в Streaming App. Аналогия: App Store developer page. Portfolio серий + metrics + subscribe.
Beat sheet
5-актная структура сезона с paywall points. Template для сценариста.
Show-centric vs Creator-centric
DramaBox — show-centric (зрители знают серию, не автора). Clipt может быть creator-centric (автор = бренд). Развилка.

11 — Streaming App

UX-паттерны стриминга

Не TikTok clone и не OTT clone [OTT = Over-the-Top, стриминг-платформа типа Netflix]. Гибрид: algo-driven discovery [алгоритмическое открытие нового контента] + show-centric retention [удержание через привязку к сериалу] + coin-gated monetization [монетизация через внутреннюю валюту].

01 Feed & Discovery
  • Personalized binge feed [персонализированная лента для запойного просмотра] — AI учится на том, что досмотрел / пропустил / пересмотрел. Обложки с hook-текстом [зацепляющим заголовком].
  • Genre hubs [разделы по жанрам] — Romance / Billionaire / Werewolf / Revenge. Navigation по вкусам, не по creators.
  • Continue watching rail [полоска «продолжить просмотр» — вернуться к сериалу в 1 нажатие]. Critical retention surface [ключевой элемент удержания].
  • Next episode autoplay [автозапуск следующего эпизода] — binge loop [петля запойного просмотра].
02 Monetization UX
  • Mid-episode paywall card [карточка оплаты посреди эпизода] — не отдельный экран, а вставка. Fade [затемнение] после клиффхэнгера → card «Unlock with 30 coins / Watch ad / Get VIP».
  • Coin wallet always visible [кошелёк монет всегда на виду] — top bar [верхняя панель]. Баланс виден в любой момент.
  • Gamified daily login [игровая механика ежедневного входа] — бесплатные coins за возврат, streak'и [серии дней подряд].
  • Free coins on signup [бесплатные монеты при регистрации] — 50-100 coins сразу. Пользователь пробует paywall без траты денег.
  • Wait timer [таймер ожидания] — ждать N часов для бесплатного unlock (как gacha [механика из мобильных игр: жди или плати]), или заплатить.
03 Onboarding
  • First play <10 секунд. Никакого signup wall [стены регистрации — когда нельзя ничего сделать без аккаунта] до опыта. Рега только при interact (like, save).
  • Taste onboarding [знакомство со вкусами — пользователь выбирает любимые жанры] — 2-3 card swipes. Calibrates feed [настраивает ленту под вкусы].
  • First episode always free + autoplay next.
  • Push notifications только после episode 3 — когда уже вложился эмоционально.
  • Daily quest introduced on day 2 — не day 1.
04 Creator Profile (App Store model)
  • Creator identity block (name, avatar, short bio, verified badge)
  • Primary metric (total views, subscribers, number of series)
  • Ongoing series with progress bar («12/40 episodes, new every Tue/Fri»)
  • Completed series (archive, sorted by popularity)
  • Behind-the-scenes / making-of reel
  • Subscribe CTA + optional tip/donate
  • Release calendar
12 — UX Insights

Что работает в AI Video UX

Работает

  • Storyboard-first workflow [сначала раскадровка, потом генерация]
  • Reference / Element library [библиотека эталонных изображений персонажей и объектов для переиспользования]
  • Natural language editing [редактирование обычным языком: «сделай пальто синим»] (Kling O1)
  • Camera presets [предустановки камеры] — pan [панорамирование], orbit [облёт], dolly [наезд/отъезд]
  • Multi-model switching [переключение между AI-моделями в одном интерфейсе]
  • Transcript-based editing [монтаж через текст — удалил строку = удалил кусок видео] (Descript)
  • Audio-to-video [загружаешь аудио → AI генерирует видео синхронно] (LTX Studio)
  • Batch 4 варианта [пакетная генерация — 4 варианта сразу, выбери лучший]

Не работает

  • Overload [перегрузка опциями] на первом экране — Krea, Runway 4.5
  • Нет aha-момента [момента «вау, это работает!»] за 40 сек — TTFV [время до первой ценности] провален у всех
  • Credit paranoia [страх потратить кредиты на неудачную генерацию]
  • Fragmented multi-tool workflow [рабочий процесс раздроблен на 6 разных тулов, в каждом свой UI]
  • Prompt engineering как барьер [для каждой модели нужен свой стиль написания запросов]
  • ComfyUI complexity [невероятно мощный, но пугающе сложный интерфейс из узлов и связей]
  • Overly "generative" chrome [интерфейс кричит «Я AI!» вместо того, чтобы быть незаметным]
  • VIP-locked settings [настройки спрятаны за платный тариф — бесит]
TTFV (Time to First Value) — 5 минут максимум, желательно <1 минуты. Если юзер не получил aha, он ушёл. Ergomania, Product Adoption Strategies 2025
